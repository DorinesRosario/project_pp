How to organize your project?

1. File structure
This paper contains a lot of good advice about project organization: http://journals.plos.org/ploscompbiol/article?id=10.1371/journal.pcbi.1000424
I use a lot of tips from that paper (some things a little bit simplified). The file structure that I use is:

2015-03-06-Retrain-ProQ2 (project folder name)
./scripts/ - directory where I store all the python/R/perl scripts
./bash/* - driver scripts that call all other scripts and execute pipelines
./bash/runall.sh - the main driver script 
./bash/filter*sh - scripts that I use usually only once to filter input directories and create soft links
./input/* - input directories
./output/* - output directories
./logs - stdout and stderr of runall.sh scripts
readme.txt - description of files and scripts in this project folder
commands.txt - commands that I run in this directory (could be a mess)

Datasets (directory for storing all datasets)
CASP9
CASP10
CASP11
Human reference genome
etc...

general_scripts (directory for storing scripts that are use in many projects)
pdb_scripts
templates
etc..

bin (all small scripts that I want to be always in PATH)
fastalen
svm_to_txt
txt_to_svm
echo_both
etc...

conf_files (directory where I store all configuration files that I use. The idea is to have them here, in addition to home folder, because I want to have them under version control.)
.vimrc 
.tmux.conf 
.inputrc
etc...

The idea of file organization is that someone not familiar with the project should be able to understand and use the scripts. This person could be anyone, but most of the time it is yourself, because you tend to forget what you did a few months ago. Also remember the Murphy's law of bioinformatics: Everything you do, you will probably have to do over again.

2. Driver (runall.sh) scripts
I attached several examples of driver scripts. Usually I name them runall.sh, but sometimes I name them based on the purpose (create-svm.sh, predict_new.sh and rosetta_energies.sh). 

Things that I find myself useful in these scripts are:
1) Usage text that describes all the parameters
2) Separate code into blocks that I call "stages". Before each stage print the date and something like
"================== stage 1 =================="
3) Always have "if [ -f $output_file ]" statement before each stage. That way if you want to rerun some stages, you can delete the corresponding output files.
4) Always print "runall.sh is running with parameters: ..." in the begining of the script. Print "runall.sh is done." in the end of the script.

3. Templates

I find useful to have templates for different types of scripts (bash, R, python, perl). In these templates I put basic usage lines, authorship comments, "script is running", "script is done." statements and similar things. I attached a few examples of templates.

4. Running scripts on Slurm.

If I have a lot of input files (such as a lot of pdb structures), I find the following way easiest to parallelize.

The driver script (such as rosetta_energies.sh) processes one single pdb file.
The slurm script (such as triolith-rosetta_energies_xargs.sh) launches the driver script many times with different input files. If you don't have many input files, then it is best if the slurm script launches driver with N input files, where N is the number of cores. However, if you have a lot of input files (like 30000 pdbs from CASP), then it is best to process more input files (let's say 1000) using one core. 

Then you can use xargs to paralelize it like this:
/bin/ls ./input_dir/*.pdb | xargs -I {} -P8 -n 1 bash -c 'runall.sh "$1" &> "$1.log"' -- {}

If your script takes more than 1 argument (not just an input file), you can do this:
/bin/ls ./input_dir/*.pdb | xargs -I {} -P8 -n 1 bash -c 'runall.sh "$1" "$2" &> "$1.log"' -- {} argument2

If you want to have stdout and stderr in separate streams, you can do this:
/bin/ls ./input_dir/*.pdb | xargs -I {} -P8 -n 1 bash -c 'runall.sh "$1" "$2" > "$1.log" 2> "$1.err"' -- {} argument2

The distributor script (such as sbatch-rosetta_energies.sh) that takes input directory and distributes all input files to the slurm script. For example, you can launch each slurm script with 1000 input files, as I mentioned above.

Check the example files, they should make sense...
